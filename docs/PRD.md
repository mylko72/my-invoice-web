# Invoice Web MVP PRD

## 1. 핵심 정보

**목적**: Notion에서 관리하는 견적서를 클라이언트가 공유 링크로 조회하고 PDF로 다운로드할 수 있도록 제공한다
**사용자**: 1인 개발자(관리자)가 발행하고 링크를 공유받는 B2B 클라이언트 담당자

---

## 2. MVP 스코프 및 목표

### MVP 범위 (포함)

- Notion 데이터베이스 연동 — 견적서 데이터 단방향 읽기
- 견적서 공개 웹 페이지 — 공유 링크(토큰)로 클라이언트 접근
- PDF 다운로드 — 브라우저 렌더링 기반 파일 생성
- 접근 보안 — 토큰/비밀번호 기반 견적서 단건 보호

### MVP 범위 (제외)

- 관리자 대시보드 및 로그인 시스템
- 견적서 직접 편집(Notion이 유일한 편집 수단)
- 이메일/SMS 발송 자동화
- 전자서명 및 결재 플로우
- 다국어 지원

---

## 3. 사용자 여정

```
[관리자] Notion DB에 견적서 작성
   ↓ Notion API → Next.js 백엔드 (캐시 포함)

1. 클라이언트: 공유 링크 수신 (이메일/메신저)
   ↓ URL 예시: /invoice/[invoiceId]?token=[accessToken]

2. 견적서 접근 페이지 진입
   ↓ 토큰 유효성 검증
   [유효] → 견적서 뷰 페이지 표시
   [무효/만료] → 접근 거부 페이지 표시
   [비밀번호 보호] → 비밀번호 입력 페이지 → 검증 후 뷰 페이지

3. 견적서 뷰 페이지
   ↓ 클라이언트 행동 선택
   [PDF 다운로드 클릭] → PDF 파일 생성 및 다운로드
   [인쇄 클릭] → 브라우저 인쇄 다이얼로그

4. 완료
   → 클라이언트 로컬에 PDF 저장
```

## 4. 기능 명세

### MVP 핵심 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|---------------|-------------|
| **F001** | Notion DB 연동 | Notion API로 견적서 데이터 조회 및 파싱 | 데이터 소스 — 이 없으면 서비스 불가 | 견적서 뷰 페이지 |
| **F002** | 견적서 웹 렌더링 | 품명/수량/단가/합계를 비즈니스 문서 레이아웃으로 표시 | 클라이언트가 실제 확인하는 핵심 기능 | 견적서 뷰 페이지 |
| **F003** | PDF 다운로드 | `견적서_[견적번호]_[발행일].pdf` 파일 생성 및 다운로드 | 클라이언트의 문서 보관·제출 요구 충족 | 견적서 뷰 페이지 |
| **F004** | 토큰 기반 접근 제어 | URL 토큰으로 견적서 단건 접근 허용/차단 | 타인의 견적서 무단 열람 방지 | 견적서 접근 페이지, 접근 거부 페이지 |

### MVP 필수 지원 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|---------------|-------------|
| **F005** | API 응답 캐싱 | Notion API 응답을 서버 사이드에서 일정 시간 캐시 | Notion API 호출 제한 및 로딩 속도 개선 | 견적서 뷰 페이지 |
| **F006** | 접근 거부/오류 처리 | 토큰 무효, 만료, 존재하지 않는 견적서 처리 | 잘못된 링크 접근 시 사용자 안내 필수 | 접근 거부 페이지 |

### MVP 이후 기능 (Phase 2)

- 관리자 로그인 및 대시보드
- 견적서 목록 조회 및 상태 관리
- 이메일 자동 발송
- 전자서명 / 수락·거절 플로우
- 비밀번호 기반 추가 보호
- 견적서 열람 이력(조회 로그)

## 6. 페이지별 상세 기능

### 견적서 접근 페이지

| 항목 | 내용 |
|------|------|
| **역할** | 공유 링크의 토큰을 검증하고 견적서 뷰 페이지로 리디렉션하는 게이트웨이 |
| **진입 경로** | 클라이언트가 공유받은 URL 직접 접근 (`/invoice/[id]?token=...`) |
| **사용자 행동** | URL 접근 즉시 자동으로 토큰 검증 처리 (별도 입력 없음) |
| **주요 기능** | - URL 쿼리 파라미터에서 토큰 추출<br>- 서버 사이드 토큰 유효성 및 견적서 ID 일치 검증<br>- 유효 시 견적서 뷰 페이지로 즉시 리디렉션<br>- 무효/만료 시 접근 거부 페이지로 리디렉션<br>- **로딩 스피너** 표시 (검증 중) |
| **다음 이동** | 토큰 유효 → 견적서 뷰 페이지, 토큰 무효/만료 → 접근 거부 페이지 |

## 8. 데이터 흐름

```
[Notion 데이터베이스]
        ↓ Notion API (서버 사이드 전용)
[Next.js Route Handler]
        ↓ 파싱 + 유효성 검증
[서버 캐시 (Next.js fetch cache / unstable_cache)]
        ↓ 캐시 HIT → 즉시 반환 / MISS → Notion API 재요청
[견적서 뷰 페이지 (Server Component)]
        ↓ props로 데이터 전달
[클라이언트 브라우저]
        ↓ PDF 다운로드 요청
[PDF 생성 (클라이언트 사이드 html2canvas + jsPDF 또는 서버 사이드 Puppeteer)]
        ↓
[로컬 파일 다운로드]
```

## 10. 보안 고려사항

### 토큰 기반 접근 제어

- 각 견적서에 Notion DB의 별도 필드로 `accessToken` (UUID v4 형식) 저장
- URL 파라미터로 전달: `/invoice/[id]?token=[accessToken]`
- 서버 사이드에서만 토큰 검증 — 클라이언트에 검증 로직 노출 금지
- 토큰은 Notion DB에서 관리자가 직접 생성/무효화

### API 키 보호

- `NOTION_API_KEY` 환경변수는 서버 사이드 전용 (`NEXT_PUBLIC_` 접두사 없음)
- Route Handler에서만 Notion API 직접 호출
- 클라이언트 번들에 API 키 노출 없음

### 기타

- 견적서 ID만으로는 접근 불가 (토큰 필수 조합)
- 민감 데이터(단가, 합계)는 토큰 검증 후에만 응답

## 11. 성능 고려사항

### Notion API 캐싱 전략

```typescript
// Next.js unstable_cache 사용 예시
import { unstable_cache } from 'next/cache'

const getInvoice = unstable_cache(
  async (invoiceId: string) => fetchFromNotion(invoiceId),
  ['invoice'],
  { revalidate: 60 } // 60초 캐시
)
```

- 기본 캐시 TTL: 60초 (변경 빈도 낮음)
- 캐시 키: `invoice-[invoiceId]`
- Notion API 무료 플랜 제한: 초당 3 요청 — 캐싱으로 충분히 대응

### 로딩 최적화

- Next.js Server Component로 초기 렌더링 (HTML 즉시 전달)
- `loading.tsx` Suspense 경계로 스켈레톤 UI 표시
- 이미지(로고 등) — `next/image` 자동 최적화

### Phase 2 — 관리자 기능

- 관리자 로그인 (Supabase Auth 또는 NextAuth)
- 견적서 목록 대시보드 (상태 필터: 발행됨 / 확인됨 / 만료)

- 견적서 조회 이력 로그 (누가 언제 열람했는지)

### Phase 3 — 클라이언트 상호작용

- 견적서 수락/거절 버튼 (이메일 알림 연동)
- 전자서명 기능
- 클라이언트 코멘트 및 수정 요청

### Phase 4 — 자동화

- 신규 Notion 항목 생성 시 자동 이메일 발송 (Webhook + Resend)
- 만기 임박 자동 리마인더
- 견적서 → 세금계산서 전환 플로우

## 17. Notion 데이터베이스 필드 정의

Notion에 다음 필드를 가진 데이터베이스를 생성해야 합니다.

| Notion 필드명 | 타입 | 설명 |
|--------------|------|------|
| Name | Title | 견적서 번호 (예: 2024-001) |
| 고객사명 | Text | 수신 고객사 이름 |
| 담당자 | Text | 고객 담당자 이름 |
| 발행일 | Date | 견적서 발행일 |
| 만기일 | Date | 견적서 유효 만기일 |
| 상태 | Select | draft / sent / accepted |
| 합계금액 | Number | 부가세 포함 최종 금액 |
| 비고 | Text | 메모 및 특이사항 |
| 접근토큰 | Text | URL 공유용 UUID 토큰 |
| 항목 | Relation or Text (JSON) | 견적 항목 목록 (별도 DB 연결 또는 JSON 저장) |

> **항목 저장 방식 권장**: 별도 `InvoiceItems` Notion 데이터베이스를 만들고 Relation으로 연결하는 방식 권장.
> 단순화를 위해 MVP에서는 항목을 JSON 텍스트 필드에 저장하는 방식도 허용.
